# Test data

These are small committed fixtures used by the pytest suite so the tests do not run the heavy Hail
tools on every invocation.
Most are generated by `workflows/create_test_data.smk`; the masked reference and the DuckDB index
golden are generated as noted under "Regenerating" below.

## Loci

Two test loci are used throughout.
chr1:100,001-200,000 is a 100 kb autosomal window.
chrX:50,000,000-50,025,000 is a 25 kb window in chrX non-PAR that exercises the haploid-male ploidy
handling in `compute-haplotypes`.

## Sample metadata

[hgdp_1kg_sample_metadata.ht](hgdp_1kg_sample_metadata.ht) holds all 4,151 HGDP+1KG samples from
`gs://gcp-public-data--gnomad/release/3.1.2/ht/genomes/gnomad.genomes.v3.1.2.hgdp_1kg_subset_sample_meta.ht`,
stripped to the key (`s`) and `gnomad_population_inference`.
[hgdp_1kg_sample_metadata.extract.ht](hgdp_1kg_sample_metadata.extract.ht) is the simplified
sample-to-population mapping produced from it by `extract-sample-metadata`.
[samples.txt](samples.txt) is the sample list used to subset the phased BCFs with bcftools.

## chr1 fixtures (chr1:100,001-200,000)

[chr1_100001_200000.ht](chr1_100001_200000.ht) holds all 9,483 gnomAD HGDP+1KG variant annotations
in the locus, subset from
`gs://gcp-public-data--gnomad/release/3.1.2/ht/genomes/gnomad.genomes.v3.1.2.hgdp_1kg_subset_variant_annotations.ht`.
[chr1_100001_200000.gnomad_afs.ht](chr1_100001_200000.gnomad_afs.ht) is the per-population
allele-frequency sites table produced from it by `extract-gnomad-afs`.
[chr1_100001_200000.vcf.gz](chr1_100001_200000.vcf.gz) (with its `.tbi`) holds the phased genotypes
for all HGDP+1KG samples in the locus.
[chr1_100001_200000_haplotypes.ht](chr1_100001_200000_haplotypes.ht) is the haplotypes table
produced by `compute-haplotypes` from the VCF, the sites table, and the sample metadata.

## chrX fixtures (chrX:50,000,000-50,025,000, non-PAR)

[chrX_50000000_50025000.ht](chrX_50000000_50025000.ht) and
[chrX_50000000_50025000.gnomad_afs.ht](chrX_50000000_50025000.gnomad_afs.ht) are the variant
annotations and per-population AF sites table for the chrX locus.
[chrX_50000000_50025000.vcf.gz](chrX_50000000_50025000.vcf.gz) (with its `.tbi`) holds the phased
genotypes for the locus.
There is deliberately no chrX haplotypes table: this locus yields zero haplotypes at the test
thresholds, so chrX serves as a sites-only contig (gnomAD variants only) in the DuckDB index tests.

## Reference FASTAs

[test.fa](test.fa) is a 1 kb single-contig (`chr1`) synthetic reference used by the `haplotype`
sequence-construction unit tests.
[test_reference.chr1_chrX.fa.gz](test_reference.chr1_chrX.fa.gz) (with `.fa.fai` and `.fa.gz.gzi`)
is a masked GRCh38 reference: every GRCh38 contig at full length, filled with `N` except the chr1
and chrX test windows, which carry real sequence.
Hail's `add_sequence` requires every reference-genome contig to be present at its full length, so a
chr1+chrX-only FASTA is rejected; the all-`N` contigs bgzip to about 12 MB, committed via git-lfs.
It is regenerated by `scripts/build_test_reference.py` (which needs the full local GRCh38 reference).

## DuckDB index golden

[duckdb_index_golden/sequences.chr1_chrX.tsv](duckdb_index_golden/sequences.chr1_chrX.tsv) is the
`sequences` table (1,306 rows, ordered by `sequence_id`) of a two-contig DuckDB index over the chr1
fixtures (haplotype + gnomAD sites) and the chrX fixture (gnomAD sites only), built against the
masked reference at `--version 9.9 --window-size 25`.
The end-to-end test `tests/tools/test_duckdb_index_e2e.py` builds the index with the per-chromosome
`init-duckdb-index` / `append-contig-to-duckdb-index` / `finalize-duckdb-index` tools and asserts
the result matches this golden row-for-row.

## Regenerating

Most fixtures are generated by the Snakemake workflow:

```bash
pixi run snakemake -j4 -s workflows/create_test_data.smk
```

The masked reference is built by `scripts/build_test_reference.py` (see its docstring).
The DuckDB index golden is produced by running the index build over the two loci with the masked
reference and exporting the `sequences` table ordered by `sequence_id`.
